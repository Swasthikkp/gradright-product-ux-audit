# 📱 Product UX/UI Audit — GradRight App (Case Study)

A hands-on product audit conducted during a talent-hunt assignment for GradRight, 
an AI-powered higher-education platform. This project showcases my approach to 
product thinking: sharp observation, structured bug reporting, and clear 
communication for cross-functional teams.

## 🎯 Objective

Evaluate the GradRight mobile app (beta) from a first-time user's perspective, 
identify friction points across UI and UX, and present findings in a way that's 
immediately actionable for engineering, design, and product teams.

## 🔍 Approach

- Explored the app end-to-end across On-Campus Programs, Online Programs, 
  Graddie AI assistant, Community, and Compare features
- Logged 10 distinct UX bugs and 40+ UI/design inconsistencies
- Prioritized findings into **Top 3 Critical Issues** rather than listing 
  everything with equal weight — quality of insight over quantity
- Documented each bug with clear reproduction steps and supporting screen 
  recordings, so any engineer could verify and fix without back-and-forth
- Annotated 40+ screenshots to visually flag UI issues (color inconsistency, 
  icon sizing, spacing, typography) with a clear legend explaining each 
  annotation type

## 🚨 Top 3 Critical Findings

| # | Issue | Impact |
|---|-------|--------|
| 1 | App exits completely on swipe-back gesture (On-Campus page) | Breaks basic navigation habit, causes immediate drop-off |
| 2 | Profile menu creates dead-ends on Community page (can't navigate away) | Traps first-time users with no clear exit |
| 3 | Zero content moderation in Community (no filters, no edit/delete) | Safety and trust risk for a student-facing platform |

*(Full list of 10 bugs with reproduction steps in `/reports/01_UX_Audit_Report.pdf`)*

## 📂 What's in this repo

- **`/reports`** — Structured PDF write-ups: UX bug log, UI audit, and product 
  improvement suggestions
- **`/screen-recordings`** — Short narrated clips demonstrating each major bug
- **`/annotated-screenshots`** — Marked-up screenshots (purple arrows/boxes/circles) 
  flagging UI issues, with a legend explaining what each annotation means

## 💭 What I'd Do Differently

Looking back, I'd tighten the language in a couple of reports — some phrasing 
leaned more dramatic than necessary. A good bug report should be precise and 
grounded, not sell the severity. I've since learned that clarity beats emphasis 
every time.
